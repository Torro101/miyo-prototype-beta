# WebView Node Bridge

The WebView bridge connects Kotlin shared state to a DOM/SVG graph and scene canvas. It should be small, typed, versioned, and replaceable.

## Goals

- Keep Kotlin as the owner of canonical project state.
- Let WebView handle pointer-heavy graph and canvas rendering in the first pass.
- Avoid arbitrary JS mutation of the project model.
- Support full snapshot rendering now and incremental patches later.
- Make bridge traffic observable for debugging.

## Transport

Platform adapters expose the same logical transport:

```text
Kotlin -> WebView: evaluate JavaScript with serialized BridgeEnvelope
WebView -> Kotlin: platform JS interface / message handler with BridgeEnvelope JSON
```

Every message uses an envelope:

```json
{
  "version": 1,
  "id": "message-id",
  "type": "GraphSnapshot",
  "replyTo": null,
  "payload": {}
}
```

Rules:

- Reject unknown major versions.
- Ignore unknown payload fields when safe.
- Include message IDs for request/response and debug logs.
- Keep payloads pure data; no JavaScript snippets as data.

## Kotlin to WebView messages

- `InitBridge`: protocol version, feature flags, theme basics.
- `GraphSnapshot`: all visible blocks, collapsed state, and derived edges.
- `SceneSnapshot`: scene objects, layout boxes, background, guides.
- `SelectionChanged`: authoritative current selection.
- `ViewportChanged`: pan/zoom or fit-to-selection request.
- `DiagnosticsChanged`: validation markers for blocks/objects/edges.
- `ApplyPatch`: future incremental update once full snapshots are stable.

## WebView to Kotlin messages

- `Ready`: DOM loaded and bridge available.
- `Select`: user selected block, edge, scene object, or empty canvas.
- `MoveBlockPreview`: transient block drag position.
- `MoveBlockCommit`: final block position command.
- `ObjectTransformPreview`: transient object layout overlay update.
- `ObjectTransformCommit`: final object layout command.
- `SetTransitionRequest`: user changed an edge/target.
- `OpenInspectorRequest`: user double-clicked or requested detailed edit.
- `ViewportChanged`: user panned/zoomed the graph or scene.
- `AssetDropRequest`: user dropped/selected an asset on the canvas.

Preview messages may update only session state. Commit messages are converted into validated editor commands.

## Graph projection

Kotlin derives a graph projection from the canonical story model:

```text
GraphNode
- blockId
- label
- color
- editorPosition
- collapsed
- diagnostics

GraphEdge
- id
- sourceBlockId
- target: block | scene | dialog
- ownerPath
- label/type
- color
- diagnostics
```

Tuesday creates line IDs from DOM element IDs and then recomputes paths from DOM geometry. Miyo should instead make edge identity stable through `ownerPath`, while WebView remains free to compute SVG path control points from current rendered geometry.

## Scene projection

```text
SceneCanvasObject
- objectId
- kind
- display asset/text/style
- layoutBox
- selectable
- diagnostics
```

The WebView may render an overlay for drag/resize/rotate. It must not rewrite model fields directly. On release, it sends `ObjectTransformCommit` with the full proposed `LayoutBox`.

## Error handling

- Malformed JSON: log and ignore.
- Unknown message type: log at debug level and ignore.
- Invalid commit payload: return `CommandRejected` with diagnostics.
- Bridge desync: Kotlin sends a fresh snapshot.
- WebView reload: WebView sends `Ready`, Kotlin replays current snapshots.

## Security and safety

- Bridge messages are data only.
- Disable external navigation by default.
- Do not grant broad filesystem access to WebView content.
- Asset URLs should be generated by the platform adapter from asset IDs.
- Custom HTML/JS legacy content, if supported later, must run in a separate constrained preview path and be flagged as unsafe/non-portable.

## First-pass implementation stance

Use full `GraphSnapshot` and `SceneSnapshot` updates after each committed command. This is simpler and adequate for the first milestone. Keep message names and IDs stable so incremental `ApplyPatch` can be added later without changing editor semantics.

## Part 6 implementation note

The first live implementation uses `host.loadGraph` as the Kotlin-to-WebView snapshot message. `MiyoProject.toGraphSnapshot()` derives graph nodes from project start, blocks, scenes, and actions, then serializes a small JSON envelope without introducing a serialization dependency yet. The bundled WebView page renders this snapshot with DOM nodes and SVG edges and emits selection, snapshot, and move-commit messages back to Kotlin. Commit messages are intentionally not persisted until the typed editor command layer lands.
